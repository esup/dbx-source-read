# DBX 后端源码分析

## 1. 模块概述

DBX 后端由两个 Rust crate 组成：

| Crate | 类型 | 代码量 | 职责 |
|-------|------|--------|------|
| `dbx-core` | 库 (lib) | ~147 个 .rs 文件，~80,000+ 行 | 核心业务逻辑：数据库连接、查询执行、Schema 管理、AI、数据传输、导出导入、Agent 管理、消息队列、Nacos 等 |
| `dbx-web` | 二进制 (bin) | ~40 个 .rs 文件，~6,000+ 行 | HTTP API 服务层：基于 Axum 的 REST API、认证鉴权、静态文件服务、SSE 推送 |

两者关系：`dbx-web` 依赖 `dbx-core`，将核心能力以 HTTP API 形式暴露给前端。`dbx-core` 同时也被 `src-tauri`（桌面版）直接调用，实现桌面端零网络开销的 IPC 模式。

## 4. 核心状态管理 — AppState

`AppState` 是后端的核心状态容器，定义在 `connection.rs`，被 `dbx-web` 和 `src-tauri` 共享：

```rust
pub struct AppState {
    pub connections: Arc<RwLock<HashMap<String, PoolKind>>>,  // 活跃连接池
    pub configs: RwLock<HashMap<String, ConnectionConfig>>,   // 连接配置
    pub running_queries: RunningQueries,                       // 运行中查询(取消用)
    pub storage: Storage,                                      // SQLite 持久化
    pub plugins: PluginRegistry,                               // 插件注册表
    pub agent_manager: AgentManager,                           // JDBC Agent 管理
    pub tunnels: TunnelManager,                                // SSH 隧道
    pub proxy_tunnels: ProxyTunnelManager,                     // TCP 代理隧道
    pub http_tunnels: HttpTunnelManager,                       // HTTP 隧道
    pub transaction_sessions: Arc<RwLock<HashMap<String, TransactionSession>>>,
    pub nacos_registry: NacosAdminRegistry,
    pub mq_registry: MqAdminRegistry,  // #[cfg(feature = "mq-admin")]
}
```

## 6. 查询执行引擎

`query.rs` (4317行) 是 SQL 查询的核心执行引擎：

### 6.1 关键常量

```rust
pub const QUERY_TIMEOUT: Duration = Duration::from_secs(30);
pub const MAX_ROWS: usize = 10000;
pub const QUERY_CANCELED: &str = "Query canceled";
```

### 6.2 核心功能

- **多语句执行**：`execute_multi` / `execute_batch` / `execute_script`
- **事务支持**：`execute_in_transaction` + `TransactionSession`
- **查询取消**：通过 `CancellationToken` + 数据库原生取消（PG cancel / MySQL kill）
- **结果序列化**：`safe_i64_to_json` 处理 JS 安全整数范围外的值
- **SQL 解析**：使用 `sqlparser` crate 进行 SQL 解析和改写
- **DuckDB 特殊处理**：draining 模式防止并发查询冲突

### 6.3 查询执行流程

```
前端请求 → query/execute → query.rs::execute_query()
    │
    ├─ 解析 SQL (sqlparser)
    ├─ 获取连接池 (AppState.connections)
    ├─ 检查可编辑性 (sql_editability)
    ├─ 设置超时 (CancellationToken + timeout)
    ├─ 路由到对应驱动 (PoolKind match)
    │   ├─ MySQL → db::mysql::query()
    │   ├─ PostgreSQL → db::postgres::query()
    │   ├─ Agent → db::agent_driver::query()
    │   └─ ...
    ├─ 结果序列化 (json_value_for_js)
    └─ 返回 QueryResult { columns, rows, affected_rows, ... }
```

## 7. 连接管理

`connection.rs` (5151行) 是最长的源文件，管理数据库连接的完整生命周期：

### 7.1 连接建立

- **多协议支持**：TCP 直连、SSH 隧道、代理隧道、HTTP 隧道
- **连接池**：每种数据库使用对应的连接池实现
- **元数据连接 vs 数据连接**：分离元数据查询和数据查询的连接配置
- **连接超时**：默认 5 秒，支持 URL 参数自定义
- **保活机制**：`keepalive_tasks` 定期 ping 防止连接空闲断开

### 7.2 ConnectionConfig

定义在 `models/connection.rs` (3074行)，包含 65+ 种数据库类型的完整连接配置：

## 20. 关键设计模式

### 20.1 双壳层共享核心

`dbx-core` 作为纯库，被 `dbx-web`（HTTP）和 `src-tauri`（IPC）共享。所有业务逻辑只实现一次，两个壳层仅负责协议适配。

### 20.2 PoolKind 枚举路由

查询执行通过 `PoolKind` 枚举 match 路由到对应数据库驱动，编译期确保穷举。

### 20.3 条件编译

- `#[cfg(feature = "duckdb-bundled")]` — DuckDB 可选
- `#[cfg(feature = "mq-admin")]` — MQ 管理可选
- `#[cfg(feature = "sqlite-sqlcipher")]` — SQLite 加密可选

### 20.4 进程隔离

两类进程隔离：
1. **JDBC Agent**：Java 子进程，stdin/stdout JSON 协议
2. **DuckDB Worker**：Rust 子进程，避免 DuckDB 阻塞主进程

### 20.5 流式取消

AI 流式响应使用 `Arc<Notify>` 注册/取消机制，支持用户随时中断。

### 20.6 下载竞速

`race_download()` 同时向 R2 CDN 和 GitHub 发起请求，取最先返回的响应。

## 22. 总结

DBX 后端是一个约 **86,000+ 行 Rust 代码** 的高性能数据库管理核心，具备以下特点：

1. **广泛的数据库支持**：原生驱动 14 种 + Agent(JDBC) 35+ 种 = 总计 65+ 种数据库
2. **分层架构**：`dbx-core`（纯逻辑）→ `dbx-web`（HTTP 壳）/ `src-tauri`（桌面壳）
3. **安全性**：Argon2 密码 + AES-GCM 加密 + SSH 隧道 + TLS + 生产安全策略
4. **可扩展性**：插件系统 + Agent 机制 + 消息队列 + Nacos 服务发现
5. **AI 集成**：多提供商（Claude/OpenAI/Gemini/DeepSeek/Qwen/Ollama）+ Agent 工具调用
6. **数据管理**：跨库传输、多格式导出导入、数据对比同步
7. **云原生**：WebDAV/Snippet 云同步、子路径部署、Docker 支持
