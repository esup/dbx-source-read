## 1. 项目速览

DBX 是一个用 **Rust + Vue 3 + Tauri 2** 构建的数据库管理工具，支持 60+ 种数据库，体积仅 ~20MB。

**一句话理解架构：**

```
用户操作 → Vue 3 前端 → Tauri IPC / HTTP API → Rust 后端 (dbx-core) → 数据库
```

**三种运行形态，同一套代码：**

| 形态 | 入口 | 后端 | 前端 |
|------|------|------|------|
| 桌面应用 | `src-tauri/` | Tauri IPC 调用 Rust | Vue 3 |
| Docker/Web | `crates/dbx-web/` | Axum HTTP 服务 | Vue 3 |
| CLI | `packages/cli/` | Node.js 直连或调 Web API | 终端输出 |

## 3. 克隆与首次构建

### 3.1 克隆仓库

```bash
git clone https://github.com/t8y2/dbx.git
cd dbx
```

### 3.2 一键启动（桌面应用开发）

```bash
make
```

这条命令会：
1. 执行 `pnpm install --frozen-lockfile` 安装前端依赖
2. 检查端口 1420 是否被占用
3. 启动 `pnpm dev:tauri`，即 Tauri 开发模式

首次编译 Rust 依赖需要较长时间（可能 10-30 分钟），后续增量编译会快很多。

### 3.3 加速开发（跳过 DuckDB）

DuckDB 编译耗时较长，日常开发可以跳过：

```bash
make dev-fast
```

这等价于 `tauri dev -- --no-default-features`，只影响本地开发，发布构建始终包含 DuckDB。

### 3.4 Web 版本开发

需要分别启动前端和后端：

```bash
# 终端 1：前端开发服务器
make dev-web

# 终端 2：后端开发服务器
make dev-backend
```

前端运行在 `http://localhost:5173`，后端运行在 `http://localhost:4224`。

### 3.5 验证构建

```bash
# 前端类型检查
pnpm typecheck

# Rust 快速检查（跳过 DuckDB）
make cargo-check-fast

# 运行测试
make test
```

## 4. 项目结构导览

### 4.1 顶层目录

```
dbx/
├── src-tauri/          ← 桌面应用壳（Tauri 入口）
├── crates/             ← Rust 核心库
│   ├── dbx-core/       ← 核心业务逻辑（最重要的 Rust 包）
│   └── dbx-web/        ← Web/Docker 后端
├── apps/
│   └── desktop/        ← Vue 3 前端应用
├── packages/           ← Node.js 工具包
│   ├── node-core/      ← 共享 Node.js 工具
│   ├── cli/            ← CLI 工具
│   └── mcp-server/     ← MCP 服务器
├── agents/             ← JDBC 代理驱动（Java/Gradle）
├── docs/               ← 文档站点（Next.js）
├── scripts/            ← 构建和发布脚本
├── deploy/             ← Docker 部署配置
├── examples/           ← 示例配置
├── Cargo.toml          ← Rust workspace 配置
├── package.json        ← Node.js 根配置
├── pnpm-workspace.yaml ← pnpm workspace 配置
└── Makefile            ← 统一开发命令
```

### 4.2 Rust 代码地图

Rust 是项目的核心，代码分三层：

```
src-tauri/src/
  main.rs              ← 程序入口（仅几行）
  lib.rs               ← Tauri 命令注册（前端调用的所有命令在这里暴露）
  data_dir.rs          ← 数据目录管理
  window_state_guard.rs ← 窗口状态

crates/dbx-core/src/   ← 核心逻辑，64 个模块
  connection.rs        ← 数据库连接管理（最大的模块之一）
  query.rs             ← 查询执行引擎
  schema.rs            ← Schema 浏览器
  storage.rs           ← 本地数据存储
  ai.rs                ← AI SQL 助手
  transfer.rs          ← 数据传输
  data_grid_sql.rs     ← 数据网格查询
  ...

crates/dbx-web/src/    ← Web 后端
  main.rs              ← Axum 服务入口
```

**关键理解：** `dbx-core` 是纯业务逻辑，不依赖任何 UI 框架。`src-tauri` 和 `dbx-web` 都是它的"壳"，分别提供桌面和 Web 接入方式。

### 4.3 前端代码地图

```
apps/desktop/src/
  main.ts              ← Vue 应用入口
  App.vue              ← 根组件（2205 行，包含主布局）
  components/          ← 830+ 个 Vue 组件
    editor/            ← SQL 编辑器（QueryEditor.vue 是核心）
    connection/        ← 连接管理（ConnectionDialog.vue 是核心）
    schema/            ← Schema 浏览器
    diff/              ← Schema 对比
    datagrid/          ← 数据网格
    admin/             ← 数据库管理面板
    export/            ← 导出功能
  composables/         ← 32 个组合式函数（业务逻辑复用）
  stores/              ← 7 个 Pinia 状态仓库
    connectionStore.ts ← 连接状态（最大，5324 行）
    queryStore.ts      ← 查询状态（3953 行）
    settingsStore.ts   ← 设置状态
  lib/                 ← 工具函数和测试
    backend/           ← 与 Rust 后端通信的封装
    __tests__/         ← 单元测试
  i18n/                ← 国际化
  styles/              ← 全局样式
```

### 4.4 前后端通信机制

**桌面模式：** 前端通过 Tauri 的 `invoke` 命令调用 Rust 函数：

```typescript
// 前端调用示例
import { invoke } from "@tauri-apps/api/core";
const result = await invoke("query_execute", { connectionId, sql });
```

Rust 侧在 `src-tauri/src/lib.rs` 中注册命令：

```rust
#[tauri::command]
async fn query_execute(connection_id: String, sql: String) -> Result<...> {
    // 调用 dbx-core 的逻辑
}
```

**Web 模式：** 前端通过 HTTP API + WebSocket 与 `dbx-web` 通信。

## 5. 核心概念理解

### 5.1 连接（Connection）

连接是 DBX 的核心概念。每个连接代表一个数据库实例的配置信息。

**关键文件：**
- Rust: `crates/dbx-core/src/connection.rs` — 连接管理逻辑
- 前端: `apps/desktop/src/stores/connectionStore.ts` — 连接状态
- 前端: `apps/desktop/src/components/connection/ConnectionDialog.vue` — 连接 UI

**连接生命周期：**
```
创建连接配置 → 保存到本地存储 → 测试连接 → 打开连接 → 执行操作 → 关闭连接
```

### 5.2 查询执行

**关键文件：**
- Rust: `crates/dbx-core/src/query.rs` — 查询引擎
- Rust: `crates/dbx-core/src/query_execution_sql.rs` — SQL 执行
- 前端: `apps/desktop/src/stores/queryStore.ts` — 查询状态
- 前端: `apps/desktop/src/components/editor/QueryEditor.vue` — 编辑器

**执行流程：**
```
用户在编辑器写 SQL
  → 前端收集 SQL 和连接信息
  → 调用 Rust 后端执行
  → 后端解析 SQL、检查风险、发送到数据库
  → 返回结果集
  → 前端在数据网格中展示
```

### 5.3 Schema 浏览

**关键文件：**
- Rust: `crates/dbx-core/src/schema.rs` — Schema 查询
- 前端: `apps/desktop/src/components/schema/` — Schema UI

每种数据库的 Schema 查询 SQL 各不相同，DBX 通过 `database_capabilities.rs` 描述各数据库的能力差异。

### 5.4 JDBC 代理系统

对于没有 Rust 原生驱动的数据库，DBX 通过 JDBC 代理扩展支持：

```
前端 → Rust (agent_manager) → Java 进程 (JDBC Agent) → 数据库
```

**关键文件：**
- Rust: `crates/dbx-core/src/agent_manager.rs` — 代理进程管理
- Rust: `crates/dbx-core/src/agent_connection.rs` — 代理连接协议
- Java: `agents/drivers/<db-type>/` — 各数据库驱动实现

### 5.5 生产安全

DBX 内置了安全检查机制，防止误操作：

- `production_safety.rs` — 危险操作拦截（DROP TABLE 等）
- `sql_risk.rs` — SQL 风险等级评估
- `sql_editability.rs` — 数据可编辑性判断

## 6. 开发工作流

### 6.1 常用 Make 命令速查

```bash
# 开发
make              # 安装依赖 + 启动桌面开发环境
make dev-fast     # 跳过 DuckDB 的快速启动
make dev-web      # 启动 Web 前端开发服务器
make dev-backend  # 启动 Web 后端开发服务器

# 检查
make check        # 运行项目检查脚本
make test         # 运行所有测试
make cargo-check-fast  # Rust 快速检查
make cargo-test-fast   # Rust 快速测试

# 构建
make build        # 类型检查 + 构建前端
make package      # 打包桌面应用

# 文档
make docs         # 启动文档站点开发服务器
```

### 6.2 代码质量工具

| 工具 | 命令 | 用途 |
|------|------|------|
| oxlint | `pnpm lint` | Vue/TS 代码检查 |
| oxfmt | `pnpm fmt` | Vue/TS 代码格式化 |
| rustfmt | `cargo fmt` | Rust 代码格式化 |
| vitest | `pnpm test` | 前端单元测试 |
| cargo test | `make cargo-test-fast` | Rust 测试 |

**Git Hooks：** 项目使用 Husky + lint-staged，提交代码时自动格式化修改的文件。

### 6.3 调试技巧

**前端调试：**
- 桌面模式下按 `F12` 或 `Ctrl+Shift+I` 打开 DevTools
- Web 模式直接使用浏览器 DevTools
- 前端有 `[STARTUP]` 日志标记启动流程

**Rust 调试：**
- 使用 `log::info!`、`log::debug!` 等宏输出日志
- 桌面模式日志输出到终端
- Web 模式日志输出到后端控制台

**前后端联调：**
- 前端调用 `invoke("xxx", ...)` 时，可在 Rust 侧对应的 `#[tauri::command]` 函数中打断点
- 使用 `log::error!` 记录异常，前端会收到错误信息

## 7. 按角色选择阅读路径

根据你的技术背景和兴趣，选择最适合的阅读路径：

### 7.1 前端开发者

**必读文件（按顺序）：**

1. `apps/desktop/src/main.ts` — 应用入口，理解启动流程
2. `apps/desktop/src/App.vue` — 根组件，理解整体布局
3. `apps/desktop/src/stores/connectionStore.ts` — 核心状态管理
4. `apps/desktop/src/stores/queryStore.ts` — 查询状态
5. `apps/desktop/src/composables/useSqlExecution.ts` — SQL 执行逻辑

7. `apps/desktop/src/components/connection/ConnectionDialog.vue` — 连接对话框

**关键技术点：**
- Vue 3 Composition API + `<script setup>`
- Pinia 状态管理
- CodeMirror 6 编辑器集成
- shadcn-vue 组件库
- Tailwind CSS 样式

### 7.2 Rust 后端开发者

**必读文件（按顺序）：**

1. `crates/dbx-core/src/lib.rs` — 模块列表，了解有哪些功能
2. `crates/dbx-core/src/connection.rs` — 连接管理（核心）
3. `crates/dbx-core/src/query.rs` — 查询引擎（核心）
4. `crates/dbx-core/src/schema.rs` — Schema 浏览
5. `crates/dbx-core/src/storage.rs` — 本地存储
6. `src-tauri/src/lib.rs` — Tauri 命令注册层

**关键技术点：**
- Tokio 异步运行时
- 各数据库驱动 API
- serde 序列化
- 错误处理（anyhow）

### 7.3 全栈/新功能开发者

建议先通读前端路径的 1-3 和后端路径的 1-3，然后：

1. 选一个简单的功能（如添加一个新的导出格式）
2. 跟踪数据流：前端组件 → Store → invoke → Tauri 命令 → dbx-core → 返回
3. 理解整条链路后开始修改

### 7.4 JDBC 代理开发者

**必读文件：**

1. `agents/docs/agent-authoring.md` — 代理编写指南
2. `agents/docs/agent-protocol-v2.md` — 代理协议 v2
3. `agents/drivers/h2/` — 参考一个简单的驱动实现
4. `crates/dbx-core/src/agent_manager.rs` — Rust 侧代理管理

## 8. 第一个任务练手

以下是由易到难的练手任务，帮你熟悉项目：

### 难度 1：修复文档错别字

1. 在 `docs/content/docs/` 中找到文档
2. 修改错别字
3. `make docs` 预览效果
4. 提交 PR

### 难度 2：添加一个前端小功能

例如：在设置页面添加一个新的开关选项。

1. 找到 `apps/desktop/src/components/config/` 下的设置组件
2. 在 `settingsStore.ts` 中添加新的状态字段
3. 添加 UI 开关组件
4. 通过 `invoke` 将设置同步到 Rust 后端

### 难度 3：修复一个标记为 `good first issue` 的 Bug

1. 访问 https://github.com/t8y2/dbx/issues
2. 筛选 `good first issue` 标签
3. 在 issue 下留言表示你要处理
4. 定位问题、修复、测试、提交 PR

### 难度 4：为现有数据库驱动添加测试

1. 选择你熟悉的数据库（如 MySQL）
2. 在 `crates/dbx-core/tests/` 中找到相关测试
3. 添加新的测试用例
4. `make cargo-test-fast` 验证

## 10. 常见问题排查

### Q: 首次 `make` 编译非常慢

**正常现象。** 首次需要编译所有 Rust 依赖（包括 DuckDB），可能需要 30 分钟以上。后续增量编译会快很多。如果想跳过 DuckDB 加速开发，使用 `make dev-fast`。

### Q: 端口 1420 被占用

Tauri 开发模式需要端口 1420。如果报错：

```bash
# 查找占用进程
lsof -nP -iTCP:1420 -sTCP:LISTEN
# 杀掉对应进程
kill <PID>
```

### Q: pnpm install 失败

```bash
# 清理缓存重试
pnpm store prune
pnpm install --no-frozen-lockfile
```

### Q: Rust 编译报 "linker not found"

- **Windows:** 确认安装了 Visual Studio Build Tools 的 C++ 组件
- **Linux:** 确认安装了 `build-essential` 和 `libssl-dev`
- **macOS:** 确认安装了 Xcode Command Line Tools：`xcode-select --install`

### Q: 前端页面空白或报错

1. 检查浏览器控制台是否有错误
2. 确认 Rust 后端已正常启动（Tauri 模式下会自动启动）
3. Web 模式下确认前端（5173）和后端（4224）都在运行

### Q: DuckDB 编译失败

DuckDB 需要 C++ 编译环境。如果不需要 DuckDB 功能，日常开发使用 `make dev-fast` 跳过。
